# JID-203 — Staff Security and Financial Boundaries

## Objective

Prove Staff role approval and privileged actions are authorized, audited, and immune to self-escalation while keeping budget and income data founder-only and absent when no real source exists.

## Business reason

Internal tooling concentrates platform authority. Unlogged role changes or broad financial visibility would create a critical governance and privacy failure.

## Constitutional sources

- Constitution Articles 4, 5, 9, 10, and 11.
- Feature Ledger `STAFF-01` and `FIN-01` in [FEATURE_LEDGER.md](../FEATURE_LEDGER.md).
- [TASK_BOARD.md](../TASK_BOARD.md).

## Prerequisites

- JID-101 quality gate evidence.
- Inventory actual role model, approval functions, grants, audit schema, emergency controls, finance sources, and every Staff/Admin payload before editing.

## Dependency gates

- JID-101: `COMPLETE`.
- Any database-write validation requires a separately authorized fresh disposable local stack.

## Scope

- Deny self-role assignment/escalation and enforce approved role-grant authority.
- Ensure privileged writes use appropriate security-definer functions, fixed search paths, least grants, and mandatory audit records.
- Test suspension/emergency controls and deny cross-role misuse.
- Enforce founder-only financial read paths where a real source exists; otherwise render no metric or surface and document the gap.
- Add focused authorization/RLS/audit tests.

## Out of scope

- Inventing budget/income data or UI, payment-provider integration, organization claim flows, unrelated Staff redesign, production changes, or broad super-admin expansion.

## Environment

- Isolated approved non-production branch/worktree; local app and explicitly authorized disposable local database only.
- No cloud Supabase, Vercel, production, `main`, shared local stack, or secrets.

## Permissions

- Staff/Admin authorization paths, focused migrations/RPCs/RLS/tests, honest finance visibility gates, and report documentation are allowed.
- Remote writes and deployment are forbidden; founder-only semantics require founder verification before release.

## Expected file domains

- Staff/Admin role approval and emergency actions, authorization helpers, audit/security-definer migrations and tests, founder-only finance guards, and `docs/command-center/reports/JID-203_REPORT.md`.

## Tests

- Self-role and unauthorized role changes fail; authorized approvals succeed and audit actor/action/target/time.
- Security-definer functions reject direct misuse and use constrained grants/search paths.
- Founder can read only real finance data; Staff/Admin and other actors receive no finance fields or placeholder metrics.
- Run focused unit/integration/RLS tests, `git diff --check`, type-check, lint, and build.

## Acceptance criteria

- No user can elevate their own role or bypass the approval hierarchy.
- Every privileged state change is authorized and auditable.
- Financial data is founder-only on the server read path; missing data produces no fabricated surface.
- Required tests pass with zero skipped security assertions.

## Evidence

- Report changed files, reused/new functions, grant/RLS matrices, audit rows, exact finance source or named absence, payload denial evidence, and command results.

## Stop conditions

- Stop if approval hierarchy or founder identity is undecidable, a real finance source is absent and implementation would require invention, remote access is required, or two repair attempts fail for the same cause.

Codex on the designated JID-203 non-production branch.

## Verifier

Independent security reviewer; founder verifies the finance boundary.
