# JID-102 — Constitutional Organization Lifecycle

## Objective

Remove every organization-facing directory-claim path and prove the constitutional sequence: verification, then deliberate creation of a separately owned Business or University Profile.

## Business reason

Directory records are platform reference data. Treating them as claimable profiles breaks identity integrity and leaves jobs, billing, and access tied to the wrong authority boundary.

## Constitutional sources

- Constitution Articles 2, 5, 6, 8, 9, 10, and 11.
- Feature Ledger `ONB-02`, `DIR-01`, `DIR-02`, `BIZ-01`, `UNI-01`, and `BILL-01` in [FEATURE_LEDGER.md](../FEATURE_LEDGER.md).
- [TASK_BOARD.md](../TASK_BOARD.md).

## Prerequisites

- JID-101 reproducible quality gate evidence is available.
- JID-107 regression evidence covers both organization profile types and staff moderation.
- Exact current routes, policies, RPCs, ownership columns, and operational anchors are inventoried before editing.

## Dependency gates

- JID-101: `COMPLETE`.
- JID-107: `COMPLETE / GREEN`. JID-106 alone does not satisfy this gate.

## Scope

- Retire public/code/copy paths for claiming a Directory record and all organization writes to Directory ownership fields.
- Preserve verification as proof of organizational representation without automatically creating or granting a Profile.
- Provide deliberate Business and University Profile creation after verification.
- Re-anchor jobs, billing, subscriptions, dashboards, and access checks to owned Profiles.
- Preserve Staff-reviewed Directory correction suggestions and full AR/EN parity.
- Add lifecycle, authorization, RLS, routing, and localization regression coverage.

## Out of scope

- Directory redesign, new actors, payment-provider work, unrelated moderation, or production data cleanup.
- Commitment scoring, university-email verification, or any claim-compatible transitional UI.

## Environment

- Work in an isolated branch/worktree from the command-center-approved base.
- Local application and an explicitly authorized fresh disposable local Supabase stack only.
- No cloud Supabase, Vercel, production, `main`, shared local stack, or secrets.

## Permissions

- Application, tests, documentation, and forward-only migrations strictly required by the lifecycle repair are allowed.
- Local database writes require explicit task authorization and loopback proof. Remote writes and deployment are forbidden.

## Expected file domains

- Organization auth/onboarding/verification, catalog correction and claim retirement, Business/University Profile creation, jobs/billing/access helpers, focused RLS and lifecycle tests, migrations if necessary, and `docs/command-center/reports/JID-102_REPORT.md`.

## Tests

- Verify → deliberate Profile creation for Business and University.
- Deny pre-verification creation, Directory writes, cross-tenant access, and all retired claim endpoints.
- Prove operational resources anchor to owned Profiles and both locales use constitutional terminology.
- Run focused tests, full authorized RLS matrix, `git diff --check`, type-check, lint, and build.

## Acceptance criteria

- No reachable UI, route, function, comment, or locale copy offers Directory claiming.
- Organizations cannot write Directory rows; correction suggestions remain Staff-reviewed.
- Verification and Profile creation are separate auditable events.
- Operational authorization uses owned Profile identity and all required gates pass with zero skips.
