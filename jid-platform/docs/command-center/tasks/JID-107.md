# JID-107 — Suspended Profile Transition Boundary

## Objective

Prevent owners from changing moderation-controlled status or content on a suspended Business or University Profile while preserving authorized, audited Staff suspend/reinstate operations.

## Business reason

A suspended organization must not bypass moderation through an owner update. The defect is proven in [JID-106_RLS_FIXTURE_SCHEMA_DRIFT_REPORT.md](../JID-106_RLS_FIXTURE_SCHEMA_DRIFT_REPORT.md).

## Constitutional sources

- Constitution Articles 2, 5, 9, 10, and 11.
- Feature Ledger `BIZ-01`, `UNI-01`, and `STAFF-01` in [FEATURE_LEDGER.md](../FEATURE_LEDGER.md).
- [TASK_BOARD.md](../TASK_BOARD.md).

## Prerequisites

- Start at the command-center-specified commit on `agent/nonprod-signup-fix`.
- Confirm clean Git state, latest migration number, absence of the next migration, current policies, tables, moderation RPCs, and audit behavior.
- Prove every database endpoint is `localhost` or `127.0.0.1` and no cloud project ref is present.

## Dependency gates

- JID-106 exact failing assertion and fixture closure evidence must be available.
- Explicit authorization for a fresh disposable local database write gate is required.

## Scope

- Create exactly one forward-only migration that repairs owner update policy for both profile tables.
- Allow permitted owner content edits only on the owner's non-suspended Profile.
- Deny owner moderation-status changes and every suspended-owner content/status update.
- Preserve Staff-only audited suspend and reinstate functions.
- Add focused Business and University regression coverage while retaining all prior 14 RLS assertions.
- Update the task report and authorized command-center state records.

## Out of scope

- Organization lifecycle redesign, cloud database changes, extra migrations, application UI, or unrelated policy cleanup.

## Environment

- Branch `agent/nonprod-signup-fix` at the specified base.
- Fresh disposable local Supabase stack on loopback with all repository migrations; preserve the shared `jid-platform` stack.
- No cloud Supabase, remote database, Vercel, production, `main`, or secrets.

## Permissions

- Exactly one repository migration, focused RLS tests, local disposable database writes, report, Task Board, and Environment Map updates are allowed.
- Remote database writes and deployment are forbidden.

## Expected file domains

- The single next migration, profile moderation RLS fixtures/tests, `docs/command-center/JID-107_SUSPENDED_PROFILE_BOUNDARY_REPORT.md`, [TASK_BOARD.md](../TASK_BOARD.md), and [ENVIRONMENT_MAP.md](../ENVIRONMENT_MAP.md).

## Tests

- Own non-suspended Business/University content update succeeds; other-owner update fails.
- Direct status change fails; suspended owner status and content changes fail for both profile types.
- Staff suspend and reinstate succeed and each is audited.
- All previous 14 plus all new RLS assertions pass with zero skips/errors.
- Run focused moderation tests, full `tests/rls`, `git diff --check`, type-check, and lint.

## Acceptance criteria

- The pre-update suspended state cannot be escaped by an owner.
- Moderation status is Staff-controlled and audited for both profile types.
- One migration only; every assertion green; disposable resources fully removed; shared stack unchanged.
- Only after green may JID-102/JID-103 be marked dependent on JID-107 `COMPLETE / GREEN`.

## Evidence

- Record loopback/project isolation, migration order, policy semantics, test identities/counts, audit rows, cleanup, changed files, and exact command results.

## Stop conditions

- Stop if endpoint isolation cannot be proven, the next migration slot is occupied, Staff audit behavior would regress, authorization is absent, or two repairs fail for the same root cause.
