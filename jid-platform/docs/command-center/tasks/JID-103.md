# JID-103 — Individual Privacy and Canonical Projection Audit

## Objective

Prove server-side owner/public/recruiter/university payload separation and make every CV rendering derive from the canonical Career Record without copying truth into a parallel model.

## Business reason

Individuals own their data. A credible professional identity requires immediate consent enforcement and one canonical source for education, experience, skills, and certifications.

## Constitutional sources

- Constitution Articles 3, 4, 5, 7, 9, 10, and 11.
- Feature Ledger `IND-01` through `IND-07`, `CV-01`, and `I18N-01` in [FEATURE_LEDGER.md](../FEATURE_LEDGER.md).
- [TASK_BOARD.md](../TASK_BOARD.md).

## Prerequisites

- JID-101 quality gate evidence.
- JID-107 green moderation/RLS baseline.
- Inventory exact Career Record storage, audience queries, visibility rules, CV APIs, and all client payloads before editing.

## Dependency gates

- JID-101: `COMPLETE`.
- JID-107: `COMPLETE / GREEN`. JID-106 alone does not satisfy this gate.

## Scope

- Separate owner, public, recruiter, and university server-computed projections.
- Enforce visibility and consent before data crosses the server boundary.
- Reconcile CV input to canonical Career Record facts; retain only legitimate ephemeral snapshot/render configuration.
- Test revocation, audience denial, missing-data omission, and AR/EN parity.
- Document Evidence Vault as a precise gap if its real source remains absent; do not fabricate it.

## Out of scope

- New social/feed mechanics, a parallel profile/CV truth store, application documents, organization lifecycle, or a speculative Evidence Vault schema.

## Environment

- Isolated approved branch/worktree; local app and explicitly authorized fresh disposable local database only.
- No cloud Supabase, Vercel, production, `main`, shared local stack, or secrets.

## Permissions

- Read-path, projection, CV-renderer, focused test, documentation, and strictly necessary authorized migration changes only.
- No remote writes or deployment.

## Expected file domains

- Individual profile queries/actions, privacy/visibility projections, CV rendering inputs, focused unit/integration/RLS tests, necessary migrations, and `docs/command-center/reports/JID-103_REPORT.md`.

## Tests

- Assert each audience receives only granted fields and revocation takes effect on the next read.
- Assert private data is absent from non-owner payloads, not client-hidden.
- Assert CV output derives from canonical Career Record fixtures and omits missing facts cleanly.
- Run focused projection/CV tests, authorized RLS tests, `git diff --check`, type-check, lint, and build.

## Acceptance criteria

- Distinct server payload contracts exist for every audience.
- No private field reaches an unauthorized client.
- CV rendering has no parallel fact source and no fabricated fallback metric/content.
- All required tests pass with zero skipped authorization assertions.

## Evidence

- Report changed files, reused/new projection functions, every dynamic field and source, denial/revocation results, CV reconciliation, gaps, and exact command evidence.

## Stop conditions

- Stop if JID-107 is not `COMPLETE / GREEN`, consent granularity is unknowable, a real source is missing and would require invention, remote access is required, or two repairs fail for the same cause.

## Executor

Codex on the designated JID-103 non-production branch.

## Verifier

Independent security reviewer, then command-center verification.

## Final-state vocabulary

Final line: `CODE_COMPLETE` or `BLOCKED_WITH_EXACT_CAUSE`.
